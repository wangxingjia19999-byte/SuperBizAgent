# 基于 RAG 的企业内部知识库问答系统设计报告

## 摘要

随着企业数字化程度不断提高，企业内部积累了大量非结构化知识资料，例如制度文档、项目文档、技术方案、运维手册、培训资料、会议纪要、FAQ、工单记录等。这些资料通常分散存储在不同系统中，存在查找效率低、知识复用困难、员工重复提问、信息更新不及时等问题。

传统关键词搜索方式只能基于词语匹配，难以理解用户问题的真实语义，也无法对多个文档中的内容进行综合归纳。大语言模型虽然具有较强的自然语言理解和生成能力，但如果直接使用通用大模型进行企业问答，容易出现知识过时、回答不准确、编造内容以及企业数据安全风险等问题。

因此，本项目设计并实现一个**基于 RAG，Retrieval-Augmented Generation，检索增强生成技术的企业内部知识库问答系统**。系统通过对企业内部文档进行解析、切分、向量化存储，并在用户提问时先检索相关知识片段，再结合大语言模型生成回答，从而提升问答的准确性、可解释性和企业知识复用效率。

# 第一章 项目背景与意义

## 1.1 项目背景

在企业日常运营中，员工经常需要查询各种内部知识，例如：

- 公司制度；
- 项目开发规范；
- API 接口文档；
- 运维部署手册；
- 常见故障处理方案；
- 人事、财务、行政流程；
- 产品说明文档；
- 客户支持知识；
- 历史项目经验。

然而，这些知识通常存在以下问题：

1. **文档分散**

   文档可能存放在飞书、钉钉、Confluence、Wiki、GitLab、网盘、本地文件服务器等多个平台，员工很难快速定位。

2. **搜索效率低**

   传统搜索主要依赖关键词匹配。如果用户不知道准确关键词，就难以找到目标文档。

3. **知识更新不及时**

   企业文档持续变化，如果知识库不同步更新，问答结果可能过时。

4. **重复咨询严重**

   很多员工会反复向同事、运维、HR 或管理员询问相同问题，浪费人力成本。

5. **大模型直接问答不可靠**

   通用大模型不了解企业内部资料，容易生成不符合企业实际情况的回答。

因此，需要构建一个能够理解企业内部文档、支持自然语言提问、可追溯答案来源的智能问答系统。

## 1.2 项目意义

本项目具有以下意义：

### 1. 提高企业知识检索效率

员工可以直接用自然语言提问，例如：

```text
公司报销流程是什么？
生产环境服务发布前需要哪些审批？
接口超时常见原因有哪些？
新人入职需要完成哪些系统权限申请？
```

系统能够自动从知识库中检索相关文档并生成回答。

### 2. 降低重复沟通成本

将企业常见问题沉淀到知识库中，减少 HR、行政、运维、技术负责人等角色的重复答疑工作。

### 3. 提升知识复用能力

项目经验、故障处理方案、技术规范等内容可以被重复利用，避免知识只掌握在少数人手中。

### 4. 提高回答准确性与可追溯性

系统回答时不仅给出答案，还可以给出引用来源，例如：

```text
参考文档：
1. 《生产发布规范》第 3.2 节
2. 《服务回滚操作手册》第 2 章
```

这样用户可以验证答案来源，降低大模型幻觉风险。

### 5. 保障企业数据安全

相比直接将全部企业数据发送给外部大模型，本系统通过权限控制、脱敏、审计等机制，提高内部知识使用的安全性。

# 第二章 需求分析

## 2.1 用户角色分析

系统主要面向以下用户角色：

| 角色 | 主要需求 |
|---|---|
| 普通员工 | 查询公司制度、流程、FAQ |
| 研发人员 | 查询技术文档、接口文档、开发规范 |
| 运维/SRE | 查询部署文档、故障处理手册、监控说明 |
| HR/行政 | 维护制度文档，减少重复答疑 |
| 管理员 | 管理用户、权限、知识库和系统配置 |
| 审计人员 | 查看用户访问记录和知识使用情况 |

## 2.2 功能需求

### 2.2.1 文档上传与管理

系统应支持企业知识文档的上传、解析和管理。

支持的文档类型包括：

| 文档类型 | 示例 |
|---|---|
| PDF | 制度文件、方案文档 |
| Word | 培训资料、操作手册 |
| Markdown | 技术文档、README |
| TXT | FAQ、说明文本 |
| HTML | Wiki 导出内容 |
| JSON/CSV | 结构化知识数据 |

主要功能：

1. 上传文档；
2. 删除文档；
3. 更新文档；
4. 查看文档状态；
5. 文档分类管理；
6. 文档权限配置；
7. 文档重新索引。

### 2.2.2 文档解析与切分

系统需要对上传文档进行解析，将文档转换为文本内容，并按照一定规则切分成较小的文本片段。

原因是：

- 大模型不能一次读取非常长的文档；
- 向量检索通常以文本块为基本单位；
- 合理切分可以提高召回准确率。

切分后每个文本块需要保留元数据，例如：

```json
{
  "document_id": "doc_001",
  "file_name": "生产发布规范.pdf",
  "chunk_id": "chunk_001",
  "page": 3,
  "section": "3.2 发布审批流程",
  "content": "生产环境发布前需要完成代码评审、测试验证和负责人审批..."
}
```

### 2.2.3 向量化与索引构建

系统应将文本块转换为向量，并存入向量数据库。

基本流程如下：

```text
文档上传
    ↓
文档解析
    ↓
文本切分
    ↓
生成 Embedding 向量
    ↓
写入向量数据库
```

向量化后的数据用于语义检索。

当用户提出问题时，系统会将问题也转换为向量，然后在向量数据库中查找语义最相近的文本块。

### 2.2.4 智能问答功能

系统的核心功能是企业知识问答。

用户输入自然语言问题后，系统需要：

1. 理解用户问题；
2. 检索相关文档片段；
3. 对检索结果进行排序；
4. 将相关内容与问题一起发送给大语言模型；
5. 生成准确、简洁、可追溯的回答；
6. 返回答案和引用来源。

示例：

用户问题：

```text
生产环境发布失败后应该怎么回滚？
```

系统回答：

```text
根据《生产发布规范》和《回滚操作手册》，生产环境发布失败后应执行以下步骤：

1. 立即暂停继续发布；
2. 通知值班 SRE 和服务负责人；
3. 根据发布单确认当前版本和上一稳定版本；
4. 执行回滚脚本；
5. 观察核心指标至少 15 分钟；
6. 确认错误率和延迟恢复正常后关闭事故单。

参考来源：
- 《生产发布规范》第 4.3 节
- 《回滚操作手册》第 2.1 节
```

### 2.2.5 多轮对话功能

系统应支持连续追问。

例如：

第一轮：

```text
生产发布前需要哪些审批？
```

第二轮：

```text
如果是紧急修复还需要这些审批吗？
```

系统应能理解第二个问题中的“这些审批”指代上一轮中的审批流程。

### 2.2.6 答案引用与溯源

为了降低大模型幻觉，系统必须提供答案来源。

每个答案应包含：

1. 引用文档名称；
2. 文档页码或章节；
3. 文档片段内容；
4. 相似度分数；
5. 可选的文档下载或查看链接。

### 2.2.7 权限控制

企业知识通常具有权限差异，例如：

| 知识类型 | 可访问角色 |
|---|---|
| 通用制度 | 全体员工 |
| 薪酬制度 | HR、管理层 |
| 生产运维文档 | 研发、SRE |
| 财务数据 | 财务部门 |
| 客户资料 | 销售、客服、管理层 |

因此系统需要：

1. 用户登录认证；
2. 角色权限控制；
3. 部门权限控制；
4. 文档级权限控制；
5. 检索时过滤无权限文档；
6. 回答时不得泄露无权限内容。

### 2.2.8 审计日志

系统应记录用户行为，包括：

1. 用户登录；
2. 文档上传；
3. 文档删除；
4. 文档检索；
5. 用户提问；
6. 系统回答；
7. 权限变更；
8. 管理员操作。

审计日志用于安全追踪和合规检查。

### 2.2.9 反馈与优化

用户可以对回答进行反馈：

| 反馈类型 | 说明 |
|---|---|
| 有帮助 | 回答准确 |
| 没帮助 | 回答不符合预期 |
| 答案错误 | 内容错误 |
| 引用错误 | 引用来源不正确 |
| 内容过时 | 文档需要更新 |

这些反馈可用于后续优化检索效果和知识库质量。

# 第三章 总体设计

## 3.1 系统设计目标

本系统设计目标如下：

1. **准确性**：通过 RAG 检索企业内部文档，减少大模型胡编乱造。
2. **可追溯性**：每个回答都应提供引用来源。
3. **安全性**：支持用户认证、权限控制、敏感信息脱敏和审计日志。
4. **可扩展性**：支持后续接入更多文档来源，例如 Wiki、飞书、钉钉、GitLab、Confluence。
5. **易维护性**：采用模块化设计，文档处理、检索、问答、权限、审计等功能分层实现。
6. **高可用性**：后续可通过缓存、异步任务、数据库优化等方式提高系统性能。

## 3.2 系统总体架构

系统整体分为以下几层：

```text
用户交互层
    ↓
API 接口层
    ↓
业务服务层
    ↓
RAG 问答核心层
    ↓
模型服务层
    ↓
数据存储层
    ↓
安全审计层
```

| 模块 | 推荐技术 |
|---|---|
| 后端框架 | FastAPI |
| 前端框架 | Vue / React |
| RAG 框架 | LangChain |
| 工作流编排 | LangGraph，可选 |

| 部署方式 | Docker / Docker Compose |
| 日志与监控 | logging / Prometheus，可选 |

## 4.2 RAG 技术原理

RAG 的核心思想是：

> 不让大模型凭空回答，而是先从知识库中检索相关资料，再让大模型基于资料生成答案。

基本流程：

```text
用户问题
    ↓
问题向量化
    ↓
向量数据库检索相关文档片段
    ↓
可选：重排序
    ↓
构造 Prompt
    ↓
大模型生成答案
    ↓
返回答案和引用来源
```

可以简单表示为：

$$
Answer = LLM(Query, Retrieve(Query, KnowledgeBase))
$$

其中：

- $$Query$$ 表示用户问题；
- $$KnowledgeBase$$ 表示企业知识库；
- $$Retrieve$$ 表示检索函数；
- $$LLM$$ 表示大语言模型；
- $$Answer$$ 表示最终回答。

## 4.3 RAG 与普通大模型问答对比

| 对比项 | 普通大模型问答 | RAG 知识库问答 |
|---|---|---|
| 是否了解企业内部资料 | 否 | 是 |
| 是否支持答案来源 | 通常不支持 | 支持 |
| 是否容易幻觉 | 较高 | 较低 |
| 是否方便更新知识 | 需要重新训练或提示 | 更新知识库即可 |
| 企业落地性 | 较弱 | 较强 |
| 权限控制 | 较难 | 可结合文档权限实现 |

# 第五章 功能模块设计

## 5.1 用户与权限管理模块

### 5.1.1 模块目标

负责用户认证、角色管理和知识访问权限控制。

### 5.1.2 主要功能

1. 用户登录；
2. JWT Token 认证；
3. 用户角色管理；
4. 部门管理；
5. 文档权限控制；
6. 查询权限校验。

### 5.1.3 角色设计

| 角色 | 权限 |
|---|---|
| Admin | 系统全部权限 |
| KnowledgeManager | 上传、更新、删除知识文档 |
| Employee | 查询有权限的知识 |
| Auditor | 查看审计日志 |
| Guest | 只读公开文档 |

## 5.2 文档管理模块

### 5.2.1 模块目标

负责企业知识文档的上传、查看、更新和删除。

### 5.2.2 功能列表

1. 上传文档；
2. 设置文档分类；
3. 设置文档权限；
4. 查看文档解析状态；
5. 重新构建索引；
6. 删除文档及其向量索引；
7. 查看文档版本历史。

### 5.2.3 文档状态设计

| 状态 | 说明 |
|---|---|
| UPLOADED | 已上传 |
| PARSING | 正在解析 |
| PARSED | 解析完成 |
| INDEXING | 正在向量化 |
| READY | 可问答 |
| FAILED | 处理失败 |

## 5.3 文档解析模块

### 5.3.1 模块目标

将不同格式的文档转换为统一的文本格式。

### 5.3.2 支持格式

| 格式 | 解析方式 |
|---|---|
| PDF | PyMuPDF / pdfplumber |
| DOCX | python-docx |
| Markdown | 直接读取 |
| TXT | 直接读取 |
| HTML | BeautifulSoup |
| CSV | pandas |

### 5.3.3 输出格式

```json
{
  "document_id": "doc_001",
  "title": "生产发布规范",
  "content": "完整解析文本",
  "metadata": {
    "file_type": "pdf",
    "author": "SRE Team",
    "created_at": "2026-04-29"
  }
}
```

## 5.4 文本切分模块

### 5.4.1 模块目标

将长文档切成适合检索和大模型输入的文本块。

### 5.4.2 切分策略

推荐使用递归字符切分策略：

```text
优先按照标题切分
其次按照段落切分
再次按照句子切分
最后按照固定长度切分
```

### 5.4.3 切分参数

| 参数 | 推荐值 |
|---|---|
| chunk_size | 500 到 1000 字 |
| chunk_overlap | 80 到 150 字 |
| top_k | 3 到 8 |
| rerank_top_k | 3 到 5 |

### 5.4.4 文本块示例

```json
{
  "chunk_id": "chunk_001",
  "document_id": "doc_001",
  "content": "生产环境发布前，必须完成代码评审、测试验证、发布审批...",
  "metadata": {
    "file_name": "生产发布规范.pdf",
    "page": 3,
    "section": "3.2 发布审批"
  }
}
```

## 5.5 向量索引模块

### 5.5.1 模块目标

将文本块转换为向量并存储到向量数据库中。

### 5.5.2 工作流程

```text
读取文本块
    ↓
调用 Embedding 模型
    ↓
生成向量
    ↓
写入向量数据库
    ↓
保存 chunk 元数据
```

### 5.5.3 向量数据内容

每条向量记录应包含：

```json
{
  "chunk_id": "chunk_001",
  "document_id": "doc_001",
  "embedding": [0.012, -0.024, 0.118],
  "content": "生产环境发布前必须完成审批...",
  "metadata": {
    "department": "SRE",
    "permission": ["sre", "developer"]
  }
}
```

## 5.6 检索模块

### 5.6.1 模块目标

根据用户问题从知识库中检索相关文本块。

### 5.6.2 检索流程

```text
用户问题
    ↓
问题向量化
    ↓
基于权限过滤文档范围
    ↓
向量相似度检索
    ↓
关键词检索补充，可选
    ↓
重排序
    ↓
返回最相关文本块
```

### 5.6.3 混合检索设计

为了提高检索效果，可采用：

```text
向量检索 + 关键词检索 + 重排序
```

例如：

| 检索方式 | 作用 |
|---|---|
| 向量检索 | 理解语义相似 |
| 关键词检索 | 精确匹配专有名词 |
| Rerank 重排序 | 提高最终上下文质量 |

## 5.7 问答生成模块

### 5.7.1 模块目标

将用户问题和检索到的上下文交给大语言模型，生成最终答案。

### 5.7.2 Prompt 设计

推荐 Prompt 模板如下：

```text
你是企业内部知识库问答助手。
请严格基于给定的参考资料回答用户问题。

要求：
1. 如果参考资料中没有答案，请回答“根据当前知识库无法确定”；
2. 不要编造不存在的信息；
3. 回答要简洁、准确、分条说明；
4. 必须给出引用来源；
5. 如果资料存在冲突，请说明冲突点。

用户问题：
{question}

参考资料：
{context}

请生成答案：
```

### 5.7.3 输出格式

```json
{
  "answer": "生产环境发布前需要完成代码评审、测试验证和负责人审批。",
  "citations": [
    {
      "document_name": "生产发布规范.pdf",
      "page": 3,
      "section": "3.2 发布审批"
    }
  ],
  "confidence": 0.86
}
```

## 5.8 多轮对话模块

### 5.8.1 模块目标

支持用户连续追问，并理解上下文。

### 5.8.2 实现方式

保存会话历史：

```json

  "messages": [

      "role": "user",

    }

```

在新问题中结合历史问题进行改写：

```text
原问题：
如果是紧急修复还需要吗？

结合上下文改写为：
生产环境紧急修复是否仍然需要代码评审、测试验证和负责人审批？
```

# 第八章 RAG 核心流程设计

## 8.1 知识入库流程

```text
管理员上传文档
    ↓
系统保存原始文件
    ↓
解析文档文本
    ↓
切分文本块
    ↓
生成 Embedding
    ↓
写入向量数据库
    ↓
保存文档和 chunk 元数据
    ↓
文档状态变为 READY
```

## 8.2 用户问答流程

```text
用户输入问题
    ↓
系统检查用户权限
    ↓
对问题进行向量化
    ↓
检索有权限访问的相关文本块
    ↓
对检索结果进行重排序
    ↓
构造 Prompt
    ↓
调用大语言模型
    ↓
生成答案
    ↓
返回答案和引用来源
    ↓
记录审计日志
```

## 8.3 无答案处理机制

如果检索不到相关内容，系统不应该强行回答，而应该返回：

```text
根据当前知识库无法确定该问题的答案。建议补充相关文档或联系知识库管理员。
```

判断条件可以包括：

1. 最高相似度低于阈值；
2. 检索结果内容与问题明显无关；
3. LLM 判断上下文不足；
4. 用户无权限访问相关文档。

# 第十章 安全设计

## 10.1 用户认证

采用 JWT Token 方式。

用户登录后，系统返回 Token。用户后续请求需要在 Header 中携带：

```http
Authorization: Bearer <token>
```

## 10.2 权限控制

权限控制应贯穿整个问答流程。

特别注意：

> 检索阶段就必须过滤无权限文档，不能先检索再只在回答时过滤。

否则可能存在敏感信息泄露风险。

## 10.3 敏感信息脱敏

系统需要对以下内容进行脱敏：

```text
手机号
邮箱
身份证号
银行卡号
密码
Token
API Key
数据库连接串
内部 IP
客户敏感信息
```

脱敏位置包括：

1. 文档入库前；
2. 发送给大模型前；
3. 日志记录前；
4. 返回前端前。

## 10.4 审计追踪

所有敏感操作需要记录审计日志。

例如：

```text
用户 A 查询了薪酬制度文档
用户 B 上传了生产数据库操作手册
管理员 C 修改了文档权限
用户 D 对回答进行了负反馈
```

# 第十六章 风险与解决方案

## 16.1 大模型幻觉风险

### 风险

大模型可能生成知识库中不存在的内容。

### 解决方案

1. Prompt 中明确要求只基于参考资料回答；
2. 检索不到资料时回答无法确定；
3. 返回引用来源；
4. 设置相似度阈值；
5. 对答案进行后处理校验。

## 16.2 文档解析不准确

### 风险

PDF 表格、扫描件、复杂排版可能解析失败。

### 解决方案

1. 使用更强的文档解析工具；
2. 对扫描件增加 OCR；
3. 保留页码和段落结构；
4. 支持人工修正文档。

## 16.3 权限泄露风险

### 风险

用户可能通过提问获取无权限内容。

### 解决方案

1. 检索前进行权限过滤；
2. LLM 输入中不包含无权限文档；
3. 审计敏感查询；
4. 对返回内容进行安全检查。

## 16.4 知识过时风险

### 风险

文档更新后索引未及时更新，导致回答过时。

### 解决方案

1. 文档版本管理；
2. 定时重新索引；
3. 用户反馈“内容过时”；
4. 标记文档更新时间。

## 16.5 检索不准确风险

### 风险

用户问题和文档表达不同，导致检索不到正确内容。

### 解决方案

1. 使用更好的 Embedding 模型；
2. 增加关键词检索；
3. 加入 Rerank 模型；
4. 优化 chunk_size 和 chunk_overlap；
5. 对问题进行改写。

# 第十八章 总结

本文设计了一个基于 RAG 的企业内部知识库问答系统。系统通过文档解析、文本切分、向量化存储、语义检索和大语言模型生成，实现对企业内部知识的智能问答。

与传统关键词搜索相比，本系统能够更好地理解用户问题，并从企业知识库中检索相关资料生成答案。与直接使用大语言模型相比，本系统通过 RAG 技术引入企业私有知识，并提供引用来源，从而提高回答的准确性和可信度。

系统整体采用模块化架构，主要包括用户权限模块、文档管理模块、文档解析模块、向量索引模块、检索模块、问答生成模块、反馈模块和审计模块。系统同时考虑了企业场景中的安全性、权限控制、审计追踪和后续扩展能力。

最终，本项目可以帮助企业提升知识检索效率，降低重复沟通成本，促进知识沉淀与复用，为企业内部智能化办公和知识管理提供有效支撑。
